进程&fork

什么是进程？

进程是程序在操作系统中的一次执行实例，是系统进行资源分配和调度的基本单位 。每个进程都有自己独立的地址空间，包括代码段、数据段、堆和栈等。CPU 的核数与可同时运行的进程数相同，若进程数超过 CPU 的核数，会分时复用 CPU 资源。

通过 fork 函数来创建进程

通过 fork 函数创建进程，

  • 父进程：fork 函数返回的是子进程 pid
  • 子进程： fork 函数返回的是 0

僵尸进程

什么是僵尸进程？

子进程通过 exit 函数传递参数值或者 main 函数的 return 退出后，这些返回值会传递给操作系统，尽管子进程执行完毕，但是操作系统不会销毁子进程，直到把这些返回值传递给子进程的父进程，处于这种状态的子进程被称为僵尸进程。因此父进程需要主动回收其创建的子进程。

僵尸进程处理方法

通过 signal 函数 + wait /waitpid 函数来接收子进程的返回值

信号处理

尽管可以通过 wait/waitpid 函数来结束子进程，但是需要父进程不断的调用 waitpid 函数或者阻塞到 wait 函数这里，这显然是不合理的，可以使用信号函数来通知父进程子进程结束，然后执行注册的信号处理来结束子进程。

signal

☀️

作用:: 注册信号处理函数

用法：

  • signo：信号类型

    信号

  • func：需要注册的信号函数指针

注意：

  1. 使用 signal 函数注册信号处理函数时，默认情况下，当信号处理函数执行完毕后，信号的处理方式会恢复为默认行为，当注册的信号函数执行完成后，下次出现相同的信号不一定会执行刚刚注册的信号函数。
  2. 发生信号执行通过 signal 注册的信号处理函数会唤醒 sleep() 阻塞的进程，sleep 函数会被中断，并且 sleep 函数会返回剩余的睡眠时间，进程不会再次自动进入sleep阻塞状态。
  3. 当 open、read、recv 等阻塞函数被信号中断时，会返回一个错误值，并且 errno 会被设置为 EINTR

相关问题

为什么需要在父进程 close(clnt_sock)，在子进程 close(serv_sock)？

关键机制： 文件描述符在内核中是引用计数的。fork() 后父子各持有一份 fd 副本，双方都关了对应的那个，真正的 socket 资源才能在没人引用时正确释放。如果忘了关：

  • 父进程 close(clnt_sock)： 父进程的职责只有一个 — 继续 accept() 新连接。跟当前这个客户端的通信完全交给子进程，父进程不需要碰 clnt_sock，所以直接关掉。如果父进程不关 clnt_sock 即使子进程关闭了客户端连接，父进程还引用着它，内核不释放，资源泄漏。
  • 子进程 close(serv_sock)：子进程的职责是处理当前这一个客户端的请求。它不需要监听新连接，留着 serv_sock 没用，反而占用资源，所以关掉。如果子进程不关 serv_sock ，监听端口一直被引用，浪费 fd，还可能带来安全问题（子进程误操作监听端口）。

通过多进程来实现客户端的 I/O 分离

通过多进程来实现客户端的 I/O 分离，分别用独立的两个进程来读取数据和发送数据。这样可以提高数据传输的效率。

  • 在 1 个进程内同时实现数据收发逻辑需要考虑更多细节。程序越复杂，这种区别越明显，这也是公认的优点。
  • 可以提高频繁交换数据的程序性能，不同进程负责输入核输出，这样无论客户端从服务端接受完数据与否都可以进行数据的传输
