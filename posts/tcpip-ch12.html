第12章 IO复用

总结

小结:: 本章主要介绍了如何使用 select 函数实现 I/O 复用

理解 I/O 复用

什么是 I/O 复用 ？

I/O：在计算器编程领域内 I/O 的含义不同于嵌入式的 I/O(引脚)，而是指的是输入 / 输出操作，对于网络编程，这可能涉及从网络套接字接收数据（输入）或向网络套接字发送数据（输出）；对于文件操作，就是从文件读取数据（输入）或者向文件写入数据（输出）。

复用：在嵌入式中，GPIO 的引脚复用是指一个物理引脚可以被配置为不同的功能，但是计算器编程领域，传统的 I/O 操作模式下，一个线程或进程通常一次只能处理一个 I/O 操作（例如一个网络连接或者一个文件的读写）。如果要处理多个 I/O 源（多个文件描述符），就需要为每个 I/O 源创建一个单独的线程或进程，这会带来较大的资源开销。select 函数允许应用程序在一个线程内同时监控多个文件描述符的状态变化，当任何一个被监控的文件描述符准备好进行 I/O 操作（例如有数据可读、可写等）时，select 函数就会返回，应用程序可以根据返回结果来决定对哪个文件描述符进行相应的 I/O 操作。这样，一个线程就可以 “复用” 来处理多个不同的 I/O 源，而不需要为每个 I/O 源创建单独的线程或进程，从而提高了资源利用率和程序的并发处理能力

多进程与 I/O 复用比较

比较项目 多进程 I/O 复用
资源隔离性 每个进程有独立地址空间，一个进程崩溃不影响其他进程，稳定性高 线程或进程内复用 I/O，若出现问题可能影响整个应用
编程复杂度 相对简单，逻辑直观，每个进程类似独立程序 复杂，需精心设计事件处理逻辑和数据结构，管理连接状态、缓冲区等
资源开销 进程创建、销毁和切换开销大，内存占用多 一个线程或进程可处理多个 I/O 事件，资源开销小
进程间 / 线程间通信 进程间通信复杂，需借助管道、消息队列、共享内存等机制 线程间通信简单，但要处理好共享资源的同步问题
可扩展性 随着并发任务数增加，性能受严重影响，可扩展性差 适合大量并发连接场景，可扩展性强
CPU 利用率 适合多核 CPU，可利用多个核心并行执行不同进程，提高计算效率 对于 CPU 密集型任务无法发挥优势，甚至因事件轮询增加额外开销，更适合 I/O 密集型任务
调试难度 相对容易定位问题 调试困难，事件驱动模型增加定位问题的难度

通过 select 函数实现 I/O 复用

方法

☀️

作用:: 监视文件描述符 1. 是否可以接收数据？ 2. 是否可以无阻塞发送数据？ 3. 是否发生异常？，如果没有发生这些事件，则进入阻塞状态，超时后返回 0

步骤：

  1. 设置文件描述符

🎨 此处原为 Excalidraw 绘图: fd_set.excalidraw

将文件描述符设置为 1 即表示监听该文件描述符



![image.png|468x380](https://uek-1314584698.cos.ap-beijing.myqcloud.com/open/img/20260530221301073.png)

  1. 使用 select 函数设置监视范围及超时时间

    • maxfd：监视对象文件描述符数量，由于文件描述符的从 3 开始递增，因此该值为最大的文件描述符 + 1
    • readset：将所有关注是否存在待读数据的文件描述符注册到 fd_set 型变量，并传递其地址值
    • writeset：将所有关注是否可传输无阻塞数据的文件描述符注册到 fd_set 型变量，并传递其地址值
    • exceptset：将所有关注是否发生异常的文件描述符注册到 fd_set 型变量，并传递其地址值
    • timeout ：调用 select 函数后为防止陷入无限阻塞状态，传递超时信息

  2. 查看结果

    调用 select 函数之后，fd_set 中的值将发生变化，原来为 1 的值是需要监视的文件描述符，这些文件描述符如果没有发生监视的事件则会变为 0，发生监视的事件则保持 1。

注意：

  1. 每次调用 select 函数之前都需重新初始化 timeval 结构体的成员变量，这是因为调用 select 函数之后 timeval 结构体的成员变量被修改为超时前的剩余时间
  2. 每次调用 select 函数之前都需要拷贝 fd_set ，保留初始值，这是因为调用 select 函数之后 fd_set 中没有发生事件的文件描述符都被初始化为 0
